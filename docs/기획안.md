# Contract Sync v2 — 서비스 기획안

> 고객 중심 프로젝트 관리 플랫폼

---

## 1. 프로젝트 개요

### 1.1 배경 및 필요성

프리랜서, 소규모 외주 업체, SI/SM 기업의 프로젝트 관리 현장에는 다음과 같은 문제가 반복된다.

| 문제 | 현장 상황 |
|------|---------|
| 업무 산재 | 엑셀, 메신저, 이메일, 구두 지시가 뒤섞여 업무가 흩어져 있음 |
| 근거 부재 | "했다/안했다" 분쟁 시 증빙할 기록이 없음 |
| 보고 부담 | 주간/월간 보고서를 매번 수작업으로 작성 |
| 수금 누락 | 업무는 완료했는데 청구를 깜빡하거나 시점을 놓침 |
| 고객 관리 부재 | 고객별 거래 이력이 머릿속에만 존재 |
| 견적 반복 | 비슷한 프로젝트인데 매번 견적서를 처음부터 작성 |

기존 프로젝트 관리 도구(Jira, Notion, Asana 등)는 내부 협업에 최적화되어 있으며, **외부 고객과의 업무 보고·피드백·수금 관리**를 하나의 흐름으로 연결하지 못한다.

### 1.2 서비스 목적

**Contract Sync v2**는 외주/용역 업무의 전체 사이클을 하나의 플랫폼에서 관리한다.

```
고객 등록 → 견적 → 계약 → 업무 수행 → 완료 보고 → 피드백 → 보고서 → 수금
```

핵심 가치 3가지:

| 가치 | 설명 |
|------|------|
| **근거 관리** | 업무 완료마다 고객에게 이메일 보고, 피드백 이력이 법적 근거가 됨 |
| **AI 자동화** | 보고서, 견적서, 문서 분석을 AI가 자동 처리하여 반복 업무 제거 |
| **고객 중심** | 고객별 프로젝트·거래·이력을 한 곳에서 추적 |

### 1.3 기대 효과

| 대상 | 효과 |
|------|------|
| PM / 실무자 | 보고서 작성 시간 80% 절감, 업무 완료 즉시 보고 가능 |
| 대표 / 경영진 | 매출·미수금·프로젝트 현황을 대시보드에서 실시간 확인 |
| 고객(발주처) | 포털에서 진행 현황 직접 확인, 이메일로 즉시 피드백 |
| 조직 전체 | "했다/안했다" 분쟁 원천 차단, 고객 신뢰 구축 |

## 2. 시장 분석

### 2.1 타겟 사용자

**주 타겟:**

| 구분 | 설명 | 규모 |
|------|------|------|
| 1순위 | IT 외주 업체 (SI/SM, 웹에이전시) | 1~30인 |
| 2순위 | 프리랜서 (개발, 디자인, 영상) | 1인 |
| 3순위 | 컨설팅/용역 업체 | 1~50인 |

**사용자 페르소나:**

**페르소나 A — "외주 PM 김과장"**
- 3~5개 외주 프로젝트를 동시 관리
- 매주 고객별 주간보고서 작성에 반나절 소모
- 엑셀로 업무 추적, 카톡으로 보고, 메일로 산출물 전달
- 수금 시점을 놓쳐 매출 인식이 늦어지는 일이 빈번

**페르소나 B — "프리랜서 이개발"**
- 2~3개 클라이언트와 동시 계약
- 견적서를 Google Sheets로 작성
- 계약서 없이 구두로 시작하는 일도 많음
- "했다/안했다" 분쟁 경험 있음

**페르소나 C — "대표 박사장"**
- 10명 규모 외주 업체 운영
- 팀원별 업무 현황 파악이 어려움
- 이번 달 매출·미수금을 정확히 모름
- 고객별 거래 이력이 정리되어 있지 않음

### 2.2 경쟁 서비스 분석

| 서비스 | 강점 | 약점 (Contract Sync 기회) |
|--------|------|-------------------------|
| **Jira** | 이슈 트래킹, 스프린트 관리 | 외부 고객 보고 불가, 수금 관리 없음, 학습 곡선 높음 |
| **Notion** | 유연한 문서 관리, 협업 | 프로젝트 관리 구조 없음, 보고서 자동화 없음 |
| **Asana** | 업무 관리, 타임라인 | 고객 포털 없음, 한국 업무 환경(HWP, 견적서) 미지원 |
| **Monday.com** | 시각적 보드, 자동화 | 외주 보고/피드백 플로우 없음, 비쌈 |
| **플로우(Flow)** | 한국형 업무 관리 | 고객 대면 보고 기능 없음, AI 없음 |
| **엑셀/시트** | 진입 장벽 없음 | 협업 불가, 자동화 없음, 이력 관리 불가 |

### 2.3 차별화 포인트

```
┌─────────────────────────────────────────────────────────┐
│               Contract Sync만의 차별점                    │
│                                                         │
│  1. 고객(발주처) 중심 설계                                │
│     → 고객별 프로젝트·거래·이력 통합 관리                   │
│                                                         │
│  2. 업무 완료 → 고객 보고 → 피드백 자동화                  │
│     → 이메일 보고 + 비로그인 피드백 + 이력 기록             │
│     → 경쟁 서비스에 없는 유일한 기능                        │
│                                                         │
│  3. AI 보고서 / 견적서 자동 생성                           │
│     → 일간/주간/월간 보고서를 AI가 작성                     │
│     → 과거 데이터 기반 견적서 초안 생성                     │
│                                                         │
│  4. 한국 업무 환경 최적화                                  │
│     → HWP 지원, Google Sheets 견적서 연동                 │
│     → 한국어 AI 보고서, 세금계산서 연동 (향후)              │
│                                                         │
│  5. 수금 관리 통합                                        │
│     → 마일스톤 완료 = 청구 알림                            │
│     → 프로젝트 관리 도구에 수금 추적은 경쟁사 부재           │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

## 3. 서비스 정의

### 3.1 서비스 한 줄 정의

> **외주/용역 프로젝트의 고객 관리부터 수금까지, AI가 보고서와 견적서를 자동 생성하는 올인원 플랫폼**

### 3.2 핵심 기능 맵

```
┌─────────────────────────────────────────────────────┐
│                 Contract Sync v2                     │
├─────────────────────────────────────────────────────┤
│                                                     │
│  [고객 관리]        [프로젝트 관리]       [업무 관리]  │
│   발주처 등록         3가지 유형           직접 생성    │
│   거래 이력          상태 관리            AI 제안     │
│   고객 포털          타임라인             담당자 지정   │
│                                        산출물 첨부   │
│                                                     │
│  [문서 관리]        [완료 보고/피드백]    [AI 보고서]   │
│   견적서(Sheets)     이메일 보고          일간/주간/월간│
│   계약서(AI 분석)    예약 전송            결과 보고서   │
│   검토 프로세스      비로그인 피드백       AI 견적서    │
│                                                     │
│  [수금 관리]        [대시보드]           [팀 협업]     │
│   결제 일정          매출 현황            RBAC        │
│   청구 알림          워크로드             댓글/멘션    │
│   수금 추적          AI 인사이트          활동 로그    │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### 3.3 서비스 전체 사이클

```
① 고객 등록
     │
     ▼
② 프로젝트 생성 ← 문서 첨부 (견적서/계약서 — 선택)
     │                    │
     ▼                    ▼
③ 업무 직접 생성     AI가 업무 제안 (보조)
     │                    │
     ├────────────────────┘
     ▼
④ 업무 진행 & 관리
     │
     ▼
⑤ 업무 완료 → 고객에게 완료 보고 이메일 (산출물 첨부)
     │                    │
     │                    ▼
     │              ⑥ 고객 피드백 수신 (확인 / 수정요청)
     │                    │
     ├────────────────────┘
     ▼
⑦ 정기 보고서 ← AI 자동 생성 (선택적 발행)
     │
     ▼
⑧ 전체 업무 완료
     │
     ▼
⑨ 결과 보고서 ← AI 자동 생성
     │
     ▼
⑩ 수금 완료 → 프로젝트 종료 → 고객 히스토리에 기록
```

## 4. 사용자 시나리오

### 4.1 시나리오 A — 외주 프로젝트 (메인 플로우)

> "웹에이전시 PM 김과장이 신규 외주 프로젝트를 수주하고 관리하는 과정"

| 단계 | 사용자 행동 | 시스템 동작 |
|------|-----------|-----------|
| 1 | 발주처 "(주)ABC" 등록 (담당자: 김대표, 이메일 입력) | clients 테이블에 저장 |
| 2 | 프로젝트 "OO시스템 구축" 생성 (유형: 외주, 발주처: ABC) | projects 생성 |
| 3 | 기존 Google Sheets 견적서 URL 연결 | documents에 연결, AI가 항목 파싱 |
| 4 | AI가 견적 항목 기반으로 업무 제안 | "화면설계", "DB설계" 등 Task 제안 |
| 5 | 원하는 업무만 선택하여 생성 + 직접 추가 업무 등록 | tasks 테이블에 INSERT |
| 6 | 계약서 업로드 → AI 분석 → 검토자 지정 → 확정 | documents + 검토 워크플로 |
| 7 | 계약서의 결제 조건 등록 (착수금 30%, 중도금 40%, 잔금 30%) | payment_schedules 생성 |
| 8 | 팀원에게 업무 배정 | assignee_id 업데이트, 알림 발송 |
| 9 | 팀원이 "화면설계" 완료 → 설계서 첨부 → 완료 처리 | task status → completed |
| 10 | "완료 보고" 모달 → 발주처에게 이메일 발송 (설계서 첨부) | completion_report 생성, 이메일 발송 |
| 11 | 발주처 김대표가 이메일에서 "확인 완료" 클릭 | task status → confirmed |
| 12 | 금요일마다 AI가 주간보고서 초안 생성 → PM이 검토 후 발송 | ai_reports 생성, 이메일 발송 |
| 13 | 설계 단계 완료 → 중도금 청구 알림 수신 | 알림: "청구 가능" |
| 14 | 모든 업무 완료 → AI 결과보고서 자동 생성 | ai_reports (completion) |
| 15 | 잔금 수금 완료 → 프로젝트 종료 | project status → completed |

### 4.2 시나리오 B — 내부 프로젝트

> "사내 신규 서비스 개발 프로젝트를 관리하는 과정"

| 단계 | 사용자 행동 | 비고 |
|------|-----------|------|
| 1 | 프로젝트 "신규 SaaS 개발" 생성 (유형: 내부, 발주처: 없음) | client_id = NULL |
| 2 | 업무 직접 등록 ("기획", "디자인", "개발", "테스트") | AI 제안 없이 직접 |
| 3 | 팀원 배정 및 진행 | 일반 업무 관리 |
| 4 | 완료 처리 시 고객 보고 없음 (내부 업무) | is_client_facing = false |

→ 발주처, 완료 보고, 피드백, 수금 기능은 사용하지 않음. 순수 업무 관리 도구로 사용.

### 4.3 시나리오 C — 유지보수 프로젝트

> "월간 유지보수 계약을 관리하는 과정"

| 단계 | 사용자 행동 | 비고 |
|------|-----------|------|
| 1 | 프로젝트 "XYZ 월간 유지보수" 생성 (유형: 유지보수) | |
| 2 | 반복 업무 설정: "월간 점검", "백업 확인" (매월 1일) | recurring_tasks |
| 3 | 매월 1일 자동으로 업무 생성됨 | 스케줄러가 자동 처리 |
| 4 | 업무 완료 후 고객에게 보고 | 월간 보고서 자동 생성 |
| 5 | 매월 유지보수비 수금 관리 | payment_schedules |

### 4.4 시나리오 D — 견적서 없이 시작되는 업무

> "구두로 의뢰받은 급한 일을 관리하는 과정"

| 단계 | 사용자 행동 | 비고 |
|------|-----------|------|
| 1 | 발주처 선택 → 프로젝트 생성 (문서 없이) | 견적서/계약서 없음 |
| 2 | 업무 직접 등록하여 작업 시작 | |
| 3 | 나중에 계약서가 만들어지면 문서 첨부 | 역순 가능 |
| 4 | 수금 조건도 나중에 등록 | |

→ **문서가 없어도 업무를 시작할 수 있는 유연한 구조**

### 4.5 시나리오 E — 발주처 담당자 관점

> "발주처 김대표가 프로젝트 현황을 확인하는 과정"

| 단계 | 행동 | 채널 |
|------|------|------|
| 1 | 업무 완료 보고 이메일 수신 | 이메일 |
| 2 | 첨부된 산출물 다운로드 | 이메일 링크 |
| 3 | "확인 완료" 또는 "수정 요청" 피드백 | 비로그인 피드백 페이지 |
| 4 | 주간 보고서 이메일 수신 | 이메일 |
| 5 | 포털에서 전체 진행률 확인 (선택) | 비로그인 포털 |

→ **발주처 담당자는 로그인이 필요 없음.** 이메일과 토큰 기반 페이지로만 상호작용.

## 5. 정보 구조 (IA)

### 5.1 엔티티 계층

```
사용자 (User)
  │
  ├── 팀 (Team) ← 멤버 관리 (RBAC)
  │
  ├── 발주처 (Client)  ← 최상위 비즈니스 엔티티
  │     │
  │     └── 프로젝트 (Project)  ← 핵심 관리 단위
  │           │
  │           ├── 업무 (Task)  ← 실행 단위
  │           │     ├── 산출물 (Attachment)
  │           │     ├── 완료 보고 (Completion Report)
  │           │     │     └── 피드백 (Client Feedback)
  │           │     └── 댓글 (Comment)
  │           │
  │           ├── 문서 (Document)
  │           │     ├── 검토 (Review)
  │           │     └── 버전 이력
  │           │
  │           ├── 보고서 (AI Report)
  │           ├── 수금 일정 (Payment Schedule)
  │           ├── 반복 업무 (Recurring Task)
  │           └── 포털 토큰 (Portal Token)
  │
  ├── 템플릿 (Project Template)
  ├── 캘린더 연동 (Calendar Sync)
  ├── 알림 (Notification)
  └── 활동 로그 (Activity Log)
```

### 5.2 페이지 구조 (사이트맵)

```
/ (랜딩 페이지)
│
├── /auth (로그인/회원가입)
│
├── /dashboard (대시보드 — 로그인 후 홈)
│     ├── 통계 카드 (6개)
│     ├── 업무 목록 (리스트/타임라인)
│     └── 비즈니스 인사이트
│
├── /clients (발주처)
│     ├── /clients (목록)
│     └── /clients/{id} (상세 — 프로젝트/거래/문서/활동)
│
├── /projects (프로젝트)
```

## 7. 기술 스택

### 7.1 기술 선정

| 구분 | 기술 | 선정 이유 |
|------|------|---------|
| 백엔드 | FastAPI (Python 3.12) | 기존 v1 유지, 비동기 고성능, AI(Gemini) 연동 편리 |
| 프론트엔드 | HTML5 + Tailwind CSS + Alpine.js | 기존 v1 유지, 경량 SPA, 학습 곡선 낮음 |
| 데이터베이스 | SQLite (비동기) | 기존 v1 유지, 소규모 서비스에 적합, 배포 간편 |
| AI | Google Gemini 2.0 Flash | 기존 v1 유지, 멀티모달(텍스트+이미지), 한국어 우수 |
| 외부 연동 | Google Sheets API v4 | 견적서 연동, 사용자 기존 습관 유지 |
| 외부 연동 | Google Calendar API | 캘린더 양방향 동기화 |
| 스케줄러 | APScheduler | 반복 업무, 예약 발송, 자동 확인 처리 |
| 배포 | Railway.app | 기존 v1 유지, 원클릭 배포 |

### 7.2 확장 고려사항

| 시점 | 고려 사항 |
|------|---------|
| 사용자 50+ | SQLite → PostgreSQL 마이그레이션 |
| 동시 접속 증가 | Redis 캐시 레이어 추가 |
| 파일 저장량 증가 | 로컬 → S3/GCS 스토리지 전환 |
| 이메일 대량 발송 | SMTP → SendGrid/SES 전환 |

## 8. 개발 로드맵

### 8.1 Phase 구성

```
Phase 0 ─── 기반 재설계 ──────────────── 핵심
  │
  ├→ Phase 1 ─── 문서 관리 ─────────── 기존 확장
  │    │
  │    ├→ Phase 2 ─── 완료 보고/피드백 ── 핵심 차별점
  │    │    │
  │    │    ├→ Phase 3 ─── AI 보고서 ──── AI 차별점
  │    │    │
  │    │    └→ Phase 4 ─── 매출/수금 ──── 비즈니스
  │    │
  │    └→ Phase 5 ─── 템플릿/반복 ───── 편의
  │
  └→ Phase 6 ─── 대시보드 ────────────── 인사이트
       │
       └→ Phase 7 ─── 포털/캘린더 ────── 확장
```

### 8.2 Phase별 상세

| Phase | 이름 | 핵심 내용 | 신규 테이블 |
|-------|------|---------|-----------|
| **0** | 기반 재설계 | DB 재설계, 발주처/프로젝트/업무 독립화, 네비게이션 | clients, projects, tasks, task_attachments |
| **1** | 문서 관리 | 문서 유형 확장, 워크플로, Google Sheets, 계약 검토 | documents, document_reviews |
| **2** | 완료 보고/피드백 | 이메일 보고, 예약 전송, 비로그인 피드백, 자동 확인 | completion_reports, client_feedbacks |
| **3** | AI 보고서 | 정기/결과 보고서 AI 생성, 견적서 AI 생성, 보고서 허브 | ai_reports |
| **4** | 매출/수금 | 결제 조건, 청구 알림, 수금 추적 | payment_schedules |
| **5** | 템플릿/반복 | 프로젝트 템플릿, 반복 업무 자동 생성 | project_templates, recurring_tasks |
| **6** | 대시보드 | 매출 차트, 워크로드, AI 인사이트 | - |
| **7** | 포털/캘린더 | 클라이언트 포털, Google Calendar/Outlook 연동 | portal_tokens, calendar_syncs, calendar_events |

### 8.3 MVP 범위

**MVP (Phase 0~2)** 로 서비스의 핵심 가치를 검증:

```
MVP에 포함:
  ✅ 발주처 관리
  ✅ 프로젝트 관리 (3가지 유형)
  ✅ 업무 관리 (독립 테이블, 직접 생성)
  ✅ 문서 관리 (업로드 + AI 분석)
  ✅ 완료 보고 이메일 (즉시/예약)
  ✅ 클라이언트 피드백 (비로그인)
  ✅ 기존 기능 유지 (인증, 팀, 댓글, 알림)

MVP에서 제외 (이후 Phase):
  ⬜ Google Sheets 연동
  ⬜ AI 보고서/견적서 자동 생성
  ⬜ 수금 관리
  ⬜ 템플릿/반복 업무
  ⬜ 대시보드 인사이트
  ⬜ 클라이언트 포털
  ⬜ 캘린더 연동
```

## 9. 리스크 및 대응 방안

| 리스크 | 영향도 | 발생 확률 | 대응 방안 |
|--------|--------|---------|---------|
| DB 마이그레이션 실패 | 높음 | 중간 | 마이그레이션 전 DB 백업 필수, 단계적 전환, 롤백 스크립트 준비 |
| Google API 제한/변경 | 중간 | 낮음 | Sheets/Calendar 연동은 후순위(Phase 1, 7), 핵심 기능과 분리 |
| AI 보고서 품질 | 중간 | 중간 | "AI가 만들고, 사람이 판단" 원칙. 반드시 사용자 검토 후 발송 |
| 이메일 발송 실패 | 높음 | 낮음 | 3회 재시도, 실패 알림, 대기열 관리 |
| 사용자 채택 저조 | 높음 | 중간 | MVP로 핵심 가치(보고/피드백)만 먼저 검증 후 확장 |
| SQLite 성능 한계 | 중간 | 낮음 | 초기 타겟이 소규모(1~30인), PostgreSQL 전환 준비 |
| 피드백 토큰 악용 | 중간 | 낮음 | IP Rate Limit, 토큰 만료, HMAC 서명 검증 |
