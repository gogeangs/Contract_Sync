## 2. 트리거 조건

### 2.1 완료 보고 모달이 열리는 조건

| 조건 | 필수 | 설명 |
|------|------|------|
| 업무 상태 → `completed` 전환 | Y | 다른 상태에서는 모달 표시 안 함 |
| `is_client_facing = true` | Y | false이면 보고 없이 바로 완료 |
| 프로젝트에 발주처가 있음 (`client_id != null`) | Y | 발주처 없으면 보고 대상 없음 |
| 발주처 연락처 이메일이 있음 (`contact_email != null`) | Y | 없으면 수신자 직접 입력 필요 |

### 2.2 모달이 열리지 않는 경우

- `is_client_facing = false` → 업무 상태만 `completed`로 전환, 끝
- 내부 프로젝트 (`project_type = internal`) → 발주처 없으므로 보고 불필요
- 발주처 이메일 미등록 → 모달은 열되, 수신자 필드 비어있음 (직접 입력)

### 2.3 완료 보고를 나중에 작성하는 경우

- 모달에서 "나중에 작성" 버튼 클릭 시:
  - 업무 상태: `completed` 유지 (아직 `report_sent` 아님)
  - 업무 상세 페이지에 "완료 보고 작성" 버튼 표시
  - 해당 버튼 클릭 시 모달 재표시

## 3. 화면별 상세 UX

### 3.1 완료 보고 작성 모달

**진입 경로:**
1. 업무 목록(칸반/리스트)에서 상태를 "완료"로 변경 → 자동 표시
2. 프로젝트 상세 > 업무 탭에서 상태를 "완료"로 변경 → 자동 표시
3. 업무 상세에서 "완료 보고 작성" 버튼 클릭 → 수동 표시

**모달 구조:**


**필드별 상세 규칙:**

| 필드 | 초기값 | 편집 | 유효성 검사 |
|------|--------|------|------------|
| 받는 사람 | `clients.contact_email` 자동 채움 | 가능 | 이메일 형식, 필수 |
| 참조(CC) | 비어있음 | 태그 입력 | 이메일 형식, 최대 10명 |
| 제목 | AI 자동 생성 | 가능 | 필수, 최대 500자 |
| 본문 | AI 자동 생성 | 가능 | 필수, 최대 10,000자 |
| 첨부파일 | 업무 산출물 자동 체크 | 체크 해제/추가 가능 | 총 50MB 이하 |
| 발송 옵션 | "지금 발송" 선택 | 가능 | 예약 시 현재 이후 시간 |

**AI 초안 생성 동작:**

1. 모달 열림과 동시에 `POST /tasks/{id}/ai-draft-report` 자동 호출
2. 로딩 중: 제목/본문 필드에 "AI가 초안을 생성하고 있습니다..." 표시 (스켈레톤)
3. 성공: 제목/본문 필드에 AI 생성 텍스트 채움
4. 실패: "AI 생성 실패. 직접 작성해 주세요." 토스트 + 빈 필드
5. [AI 생성] 버튼 클릭 시 재생성 가능 (이미 편집한 내용 덮어쓰기 전 확인)

**발송 버튼 동작:**

| 옵션 | 동작 | 업무 상태 변경 |
|------|------|-------------|
| 지금 발송 | 즉시 이메일 전송 | completed → report_sent → feedback_pending |
| 예약 발송 | `status='scheduled'` 저장 | completed 유지 (발송 시 변경) |
| 나중에 작성 | 모달 닫기, 저장 안 함 | completed 유지 |

**피드백 유형별 동작:**

| 유형 | 클릭 동작 | 입력 필드 | 제출 후 업무 상태 | 알림 대상 |
|------|---------|---------|---------------|---------|
| 확인 완료 | 확인 다이얼로그 "확인 완료 처리하시겠습니까?" | 없음 | → `confirmed` | 담당자 |
| 수정 요청 | textarea 표시 (수정 사유 필수) | 사유 (필수, 1~1000자) | → `revision_requested` | 담당자 + 팀 전체 |
| 의견 남기기 | textarea 표시 (의견 선택) | 이름(선택) + 의견(1~1000자) | 변경 없음 | 담당자 |

**재접속 시 동작:**
- 이미 피드백을 제출한 경우에도 재접속 가능
- 이전 피드백 이력이 표시됨
- 추가 피드백 제출 가능 (수정 요청 후 재확인 등)
- "확인 완료" 후에도 "수정 요청"으로 번복 가능 → 업무 상태 덮어쓰기

**Rate Limit:**
- IP 기반 10회/분 (토큰 인증이 아닌 비로그인이므로 남용 방지)

### 3.4 클라이언트 포털과의 연결

클라이언트 포털(`/portal/{token}`)은 별도 프로세스(Step 9)이지만, 피드백 탭에서 이 프로세스와 연결됩니다:

- 포털 > "피드백 대기" 탭: `feedback_pending` 상태인 업무 목록
- 각 업무에 "피드백 보내기" 버튼 → `/feedback/{feedback_token}`으로 이동
- 포털 토큰과 피드백 토큰은 별도 (포털은 프로젝트 단위, 피드백은 보고 단위)

## 4. 상태 전이도

### 4.1 업무(Task) 상태 전이 — 완료 보고 경로

```
                    ┌──────────────────────────────────────────────┐
                    │              일반 업무 경로                    │
                    │  pending → in_progress → completed (종료)     │
                    └──────────────────────────────────────────────┘

                    ┌──────────────────────────────────────────────┐
                    │         발주처 대면 업무 경로                   │
                    │                                              │
  pending ──→ in_progress ──→ completed ──→ report_sent            │
                    ▲                              │               │
                    │                              ▼               │
                    │                       feedback_pending       │
                    │                        │         │           │
                    │                  ┌─────┘         └────┐      │
                    │                  ▼                     ▼      │
                    │           confirmed              revision    │
                    │           (종료)                 _requested   │
                    │                                     │        │
                    └─────────────────────────────────────┘        │
                     (수정 후 재작업)                               │
                    └──────────────────────────────────────────────┘
```

### 4.2 완료 보고(CompletionReport) 상태 전이

```
  draft ──→ scheduled ──→ sent
    │                      ▲
    └──────────────────────┘
              (즉시 발송)

  scheduled ──→ (취소) → 삭제
  sent ──→ failed (발송 실패, 3회 재시도 후)
```

### 4.3 상태 변경 시점 정리

| 이벤트 | Task 상태 | Report 상태 | 시점 |
|--------|----------|------------|------|
| 업무 완료 전환 | → `completed` | — | 사용자 액션 |
| 보고 작성 (즉시 발송) | → `report_sent` → `feedback_pending` | → `sent` | 사용자 액션 |
| 보고 작성 (예약 발송) | `completed` 유지 | → `scheduled` | 사용자 액션 |
| 예약 시간 도래 | → `report_sent` → `feedback_pending` | → `sent` | 시스템 (APScheduler) |
| 발송 실패 | `completed`로 롤백 | → `failed` | 시스템 |
| 피드백: 확인 완료 | → `confirmed` | — | 발주처 액션 |
| 피드백: 수정 요청 | → `revision_requested` | — | 발주처 액션 |
| 피드백: 의견 | 변경 없음 | — | 발주처 액션 |
| 7일 무응답 | → `confirmed` | — | 시스템 (APScheduler) |
| 수정 후 재작업 시작 | → `in_progress` | — | 사용자 액션 |

## 6. 이메일 발송 규칙

### 6.1 완료 보고 이메일

| 항목 | 규칙 |
|------|------|
| 발신자 | 시스템 SMTP (설정된 발신 주소) |
| 수신자 | `recipient_email` (필수) |
| 참조 | `cc_emails` (선택, 최대 10명) |
| 제목 형식 | `[프로젝트명] "업무명" 완료 안내` |
| 본문 | HTML (Jinja2 렌더링) |
| 하단 고정 | 피드백 링크 버튼 + 자동확인 안내 문구 |
| 첨부파일 | 선택된 산출물 + 추가 파일 (총 50MB 이하) |
| 재시도 | 실패 시 3회 (5분 간격) |
| Rate Limit | 사용자당 10회/분 |

### 6.2 리마인더 이메일

| 항목 | 규칙 |
|------|------|
| 트리거 | `feedback_pending` 상태 + 발송 후 4일 경과 (D+4) |
| 수신자 | 원본 보고의 `recipient_email` |
| 제목 | `[리마인더] "업무명" 완료 확인 요청` |
| 본문 | 자동 확인 예정일(D+7) 안내 + 피드백 링크 |
| 발송 조건 | 아직 피드백이 없는 경우만 |

### 6.3 CAN-SPAM 준수

- 이메일 하단에 서비스명 표시
- 수신거부 안내 문구 포함

## 7. Background 작업 (APScheduler)

### 7.1 예약 발송 체크

| 항목 | 값 |
|------|------|
| 주기 | 1분 간격 |
| 조건 | `status='scheduled'` AND `scheduled_at <= now()` |
| 동작 | 이메일 발송 → `status='sent'` → Task 상태 변경 |
| 실패 시 | 3회 재시도 (5분 간격), 최종 실패 시 `status='failed'` + 알림 |

### 7.2 자동 확인 체크

| 항목 | 값 |
|------|------|
| 주기 | 1일 1회 (매일 00:00 UTC) |
| 조건 | Task `status='feedback_pending'` AND 보고 `sent_at + 7일 <= now()` |
| 동작 | Task `status → confirmed` + activity_log 기록 |
| 활동 로그 | action: `auto_confirm`, detail: "시스템에 의해 자동 확인 처리됨" |

### 7.3 리마인더 발송

| 항목 | 값 |
|------|------|
| 주기 | 1일 1회 (매일 09:00 KST) |
| 조건 | Task `status='feedback_pending'` AND 보고 `sent_at + 4일 <= now()` AND `sent_at + 5일 > now()` AND 피드백 없음 |
| 동작 | 리마인더 이메일 발송 |

## 8. 에러 처리 및 엣지 케이스

### 8.1 이메일 발송 실패

| 상황 | 처리 |
|------|------|
| SMTP 연결 실패 | 3회 재시도 (5분 간격) → 최종 실패 알림 |
| 잘못된 이메일 주소 | 즉시 `failed` + "수신자 이메일을 확인해 주세요" 알림 |
| 첨부파일 용량 초과 (서버) | 400 에러 + "첨부파일 총 용량이 50MB를 초과합니다" |
| 발송 실패 후 재발송 | "재발송" 버튼으로 수동 재시도 가능 |

### 8.2 토큰 관련

| 상황 | 처리 |
|------|------|
| 토큰 만료 (30일 경과) | "이 링크는 만료되었습니다. 담당자에게 문의해 주세요." |
| 잘못된 토큰 | 404 페이지 |
| 동일 토큰 다중 피드백 | 허용 (이력 누적, 최신 피드백이 상태 결정) |

### 8.3 동시성

| 상황 | 처리 |
|------|------|
| 피드백 제출과 자동 확인 동시 | 피드백 우선 (DB 트랜잭션으로 보호) |
| 동일 업무에 중복 보고 작성 | 기존 보고가 있으면 "이미 발송된 보고가 있습니다. 재발송하시겠습니까?" 확인 |
| 예약 발송 중 보고 수정 | `scheduled` 상태에서만 수정/취소 가능 |

### 8.4 수정 요청 후 재작업 플로우

```
업무 상태: revision_requested
  ↓
담당자가 "진행중으로 변경" 클릭
  ↓
업무 상태: in_progress (기존 보고/피드백 이력 유지)
  ↓
수정 작업 수행
  ↓
업무 상태: completed로 변경
  ↓
완료 보고 모달 재표시 (이전 보고 내용 프리필)
  ↓
재발송 → report_sent → feedback_pending (새 사이클)
```
