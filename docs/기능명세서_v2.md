# Contract Sync v2 - 기능 명세서

> 고객 중심 프로젝트 관리 플랫폼 — AI 기반 문서 분석, 업무 관리, 완료 보고 및 비즈니스 인사이트

---

## 1. 시스템 개요

### 1.1 서비스 목적

Contract Sync v2는 프리랜서 및 소규모 외주 업체가 **발주처(클라이언트)별로 프로젝트를 통합 관리**하고, AI가 반복 업무(보고서 작성, 견적서 생성, 문서 분석)를 자동화하는 플랫폼이다.

**v1 → v2 핵심 전환:**

| 구분 | v1 | v2 |
|------|----|----|
| 서비스 중심 | 계약서 (문서) | 발주처 + 업무 |
| 업무 생성 | 계약서에서 자동 생성 | 직접 생성 + AI 보조 (선택) |
| 문서 역할 | 필수 (시작점) | 선택 (첨부/참고) |
| 대상 업무 | 외주 계약만 | 외주 + 내부 + 유지보수 |
| 문서 유형 | 계약서만 | 견적서, 계약서, 제안서 등 |
| 발주처 관리 | 없음 | 발주처별 이력/대시보드 |
| 업무 완료 | 상태 변경만 | 고객 보고 + 피드백 수신 |
| 보고서 | 없음 | AI 자동 생성 (정기 + 결과) |
| 수금 관리 | 없음 | 마일스톤별 청구/수금 추적 |

### 1.2 기술 스택

| 구분 | 기술 | 용도 |
|------|------|------|
| 백엔드 | FastAPI (Python 3.12) | 비동기 REST API 서버 |
| 프론트엔드 | HTML5 + Tailwind CSS + Alpine.js 3.x | SPA (Single Page Application) |
| 데이터베이스 | SQLite (aiosqlite, SQLAlchemy 비동기) | 로컬 데이터 저장 |
| AI 엔진 | Google Gemini 2.0 Flash (멀티모달) | 문서 분석, 보고서/견적서 생성 |
| 인증 | 이메일 인증 + Google OAuth 2.0 | 사용자 인증 |
| 파일 처리 | PyMuPDF, python-docx, olefile, Pillow | 다양한 문서 포맷 파싱 |
| 외부 연동 | Google Sheets API v4 | 견적서 연동 |
| 외부 연동 | Google Calendar API / Outlook REST API | 캘린더 양방향 동기화 |
| 이메일 | aiosmtplib + Jinja2 템플릿 | 완료 보고, 정기 보고 발송 |
| 웹폰트 | Noto Sans KR | 한글 UI 렌더링 |

### 1.3 전체 사용자 흐름

```
[비로그인] 랜딩 페이지 → 회원가입/로그인
    ↓
[로그인 후] 대시보드 (통합 현황)
    ├── 발주처 관리 → 발주처 등록 / 상세(거래 이력)
    ├── 프로젝트 관리 → 생성(직접/문서기반/템플릿) / 상세
    │     ├── 업무 관리 → 생성 / 상태 변경 / 담당자 지정 / 산출물 첨부
    │     ├── 문서 관리 → 업로드 / AI 분석 / 검토 / 확정
    │     ├── 완료 보고 → 이메일 발송 / 예약 / 피드백 수신
    │     ├── 보고서 → 정기(일간/주간/월간) / 결과 보고서
    │     └── 수금 관리 → 결제 조건 / 청구 / 수금 추적
    ├── 팀 관리 → 팀 생성 / 멤버 초대 / 역할 관리
    └── 설정 → 다크 모드 / 캘린더 연동 / 이메일 서명
```

### 1.4 용어 정의

| 용어 | 정의 |
|------|------|
| 발주처 (Client) | 업무를 의뢰하는 외부 고객/기관 |
| 프로젝트 (Project) | 하나의 사업 단위. 기존 "계약"을 포함하는 상위 개념 |
| 업무 (Task) | 프로젝트 내 실행 단위. 독립 생성 가능 |
| 문서 (Document) | 견적서/계약서/제안서 등 프로젝트에 첨부되는 파일 |
| 산출물 (Attachment) | 업무 완료의 근거가 되는 첨부 파일 |
| 완료 보고 (Completion Report) | 업무 완료 시 발주처에게 발송하는 이메일 보고 |
| 피드백 토큰 (Feedback Token) | 비로그인 발주처 피드백을 위한 일회용 접근 키 |
| 정기 보고서 (Periodic Report) | AI가 자동 생성하는 일간/주간/월간 보고서 |
| 결과 보고서 (Completion Report) | 프로젝트 완료 시 AI가 생성하는 종합 보고서 |

### 1.5 엔티티 계층 구조

```
발주처 (Client)           ← 최상위 (선택)
  └─ 프로젝트 (Project)   ← 핵심 단위
       ├─ 업무 (Task)     ← 실행 단위
       │   ├─ 산출물 (Attachment)
       │   ├─ 완료 보고 (Completion Report)
       │   │   └─ 피드백 (Client Feedback)
       │   └─ 댓글 (Comment)
       ├─ 문서 (Document)
       │   ├─ 검토 (Document Review)
       │   └─ 버전 이력
       ├─ 보고서 (AI Report)
       ├─ 수금 일정 (Payment Schedule)
       └─ 반복 업무 설정 (Recurring Task)
```

## 2. 사용자 인증

### 2.1 이메일 회원가입

**사용자 흐름:**

1. **이메일 입력 단계**
   - 사용자가 회원가입 버튼 클릭 → 로그인 모달이 열림
   - 이메일 주소를 입력하고 "인증코드 발송" 버튼 클릭
   - 서버에서 6자리 숫자 인증코드를 생성하여 SMTP 메일 발송
   - 인증코드 유효 시간: 10분
   - 이미 가입된 이메일인 경우 오류 메시지 표시

2. **인증코드 검증 단계**
   - 화면이 인증코드 입력 폼으로 전환됨
   - 6자리 숫자를 입력하면 자동으로 검증 요청 전송
   - 성공 시 비밀번호 설정 화면으로 전환
   - 실패 시 "잘못된 인증코드입니다" 오류 표시
   - "인증코드 재발송" 링크로 새 코드 요청 가능

3. **비밀번호 설정 단계**
   - 비밀번호 입력 필드 + 확인 필드
   - **비밀번호 강도 미터**: 실시간으로 강도를 시각 표시
     - 약함 (빨간색): 기본 조건만 충족
     - 보통 (노란색): 대소문자 + 숫자 혼합
     - 강함 (초록색): 12자 이상 + 특수문자 포함
   - 비밀번호 표시/숨기기 토글 (눈 아이콘)
   - 비밀번호 규칙: 8자 이상, 영문 + 숫자 필수
   - 가입 완료 시 자동 로그인 후 대시보드로 이동

**비밀번호 강도 점수 산정:**

| 조건 | 점수 |
|------|------|
| 8자 이상 | +1 |
| 12자 이상 | +1 |
| 대문자 포함 | +1 |
| 소문자 포함 | +1 |
| 숫자 포함 | +1 |
| 특수문자 포함 | +1 |

- 0~2점: 약함 (빨간색 바)
- 3~4점: 보통 (노란색 바)
- 5~6점: 강함 (초록색 바)

### 2.2 이메일 로그인

**사용자 흐름:**
1. 이메일 + 비밀번호 입력
2. 비밀번호 표시/숨기기 토글 지원
3. 로그인 성공 시 세션 쿠키 설정 후 대시보드 이동
4. 실패 시 "이메일 또는 비밀번호가 올바르지 않습니다" 표시

**보안 처리:**
- bcrypt 솔트 기반 해싱으로 비밀번호 저장
- 사용자가 존재하지 않는 경우에도 더미 해시를 검증하여 타이밍 공격 방지
- 세션 토큰: HttpOnly 쿠키, 24시간 유효, SameSite=Lax

### 2.3 Google OAuth 로그인

**사용자 흐름:**
1. "Google로 로그인" 버튼 클릭
2. Google 인증 페이지로 리다이렉트 (openid, email, profile 범위)
3. Google 인증 완료 후 콜백 URL로 리다이렉트
4. 기존 사용자: 세션 생성 후 대시보드 이동
5. 신규 사용자: 자동 가입 (Google 이름/프로필 사진 저장) 후 대시보드 이동

### 2.4 로그아웃
- 헤더의 프로필 아이콘 드롭다운에서 "로그아웃" 클릭
- 서버 세션 삭제 + 쿠키 제거
- 랜딩 페이지로 이동

### 2.5 세션 관리

| 항목 | 상세 |
|------|------|
| 저장 방식 | DB (user_sessions 테이블) |
| 토큰 형식 | URL-safe 32바이트 랜덤 문자열 |
| 유효 기간 | 24시간 |
| 쿠키 속성 | HttpOnly, SameSite=Lax, Secure(프로덕션) |
| 만료 세션 정리 | 1시간 주기 자동 삭제 |

### 2.6 API 엔드포인트

| 메서드 | 경로 | 설명 | Rate Limit |
|--------|------|------|------------|
| POST | `/api/v1/auth/send-code` | 인증코드 발송 | 5회/분 |
| POST | `/api/v1/auth/verify-code` | 인증코드 검증 | 5회/분 |
| POST | `/api/v1/auth/signup` | 회원가입 | 3회/분 |
| POST | `/api/v1/auth/login/email` | 이메일 로그인 | 3회/분 |
| GET | `/api/v1/auth/login/google` | Google 로그인 시작 | - |
| GET | `/api/v1/auth/callback/google` | Google 콜백 처리 | - |
| GET | `/api/v1/auth/me` | 현재 사용자 조회 (팀 목록 포함) | - |
| PATCH | `/api/v1/auth/profile` | 프로필 수정 | - |
| POST | `/api/v1/auth/logout` | 로그아웃 | - |

## 3. 발주처(클라이언트) 관리

### 3.1 개요

발주처는 시스템의 **최상위 엔티티**로, 프로젝트의 업무 의뢰 주체를 나타낸다. 내부 프로젝트는 발주처 없이 생성 가능하다 (client_id = NULL).

### 3.2 발주처 등록

**필수 필드:** 발주처명
**선택 필드:** 담당자명, 담당자 이메일, 담당자 전화번호, 주소, 업종/분류, 메모

**데이터 모델 (clients 테이블):**

| 필드 | 타입 | 설명 | 제약조건 |
|------|------|------|---------|
| id | INTEGER, PK | 발주처 ID | AUTOINCREMENT |
| user_id | INTEGER, FK | 생성자 | users.id, NOT NULL |
| team_id | INTEGER, FK | 팀 소속 | teams.id, NULLABLE |
| name | VARCHAR(200) | 발주처명 | NOT NULL |
| contact_name | VARCHAR(100) | 담당자명 | NULLABLE |
| contact_email | VARCHAR(200) | 담당자 이메일 | NULLABLE, 이메일 형식 검증 |
| contact_phone | VARCHAR(20) | 담당자 전화번호 | NULLABLE |
| address | TEXT | 주소 | NULLABLE |
| category | VARCHAR(50) | 업종/분류 | NULLABLE |
| memo | TEXT | 메모 | NULLABLE, 최대 2000자 |
| created_at | DATETIME | 생성일 | UTC |
| updated_at | DATETIME | 수정일 | UTC |

**중복 검사:** 동일 팀(또는 개인) 범위 내에서 발주처명 중복 불가.

### 3.3 발주처 목록 / 검색

- 카드형 목록 (최신순 정렬)
- 검색: 발주처명, 담당자명 (300ms 디바운스)
- 필터: 업종/분류
- 각 카드 표시 정보:
  - 발주처명, 담당자명
  - 진행중 프로젝트 수
  - 총 거래액
  - 최근 거래일

### 3.4 발주처 상세 (발주처 대시보드)

- **상단:** 발주처 기본 정보 + 편집 버튼
- **프로젝트 목록:** 해당 발주처의 전체 프로젝트 (진행중/완료 탭)
- **거래 이력:** 프로젝트별 계약 금액, 수금 현황 요약
- **문서 이력:** 해당 발주처와 관련된 모든 문서 (견적서, 계약서 등)
- **최근 활동 로그**

### 3.5 발주처 수정 / 삭제

- **수정:** 기본 정보 인라인 편집
- **삭제:** 연관 프로젝트가 있는 경우 삭제 불가 (경고 메시지 표시)
- 삭제 시 확인 다이얼로그 (빨간색 위험 모드)

### 3.6 API 엔드포인트

| 메서드 | 경로 | 설명 | Rate Limit |
|--------|------|------|------------|
| POST | `/api/v1/clients` | 발주처 등록 | 20회/분 |
| GET | `/api/v1/clients` | 발주처 목록 (page, size, search, category) | - |
| GET | `/api/v1/clients/{id}` | 발주처 상세 (프로젝트/거래 포함) | - |
| PUT | `/api/v1/clients/{id}` | 발주처 수정 | 20회/분 |
| DELETE | `/api/v1/clients/{id}` | 발주처 삭제 | - |
| GET | `/api/v1/clients/{id}/projects` | 발주처의 프로젝트 목록 | - |
| GET | `/api/v1/clients/{id}/documents` | 발주처의 문서 목록 | - |

## 4. 프로젝트 관리

### 4.1 개요

기존 "계약(Contract)"을 "프로젝트(Project)"로 확장한다. 프로젝트는 업무, 문서, 보고서, 수금의 상위 컨테이너이며, 발주처와의 연결은 선택사항이다.

### 4.2 프로젝트 유형

| 유형 | 코드 | 발주처 | 설명 |
|------|------|--------|------|
| 외주 | outsourcing | 필수 | 외부 클라이언트의 용역 |
| 내부 | internal | 없음 | 자체 프로젝트 (R&D, 내부 도구 등) |
| 유지보수 | maintenance | 선택 | 정기 유지보수. 반복 업무 지원 |

### 4.3 프로젝트 생성

**생성 방법 3가지:**

| 방법 | 설명 | UI 진입점 |
|------|------|---------|
| 직접 생성 | 프로젝트명, 유형, 발주처(선택), 기간, 설명 수동 입력 | 대시보드 "새 프로젝트" 버튼 |
| 문서 기반 생성 | 계약서/견적서 업로드 → AI 분석 → 프로젝트 자동 구성 | 문서 업로드 화면 |
| 템플릿 기반 생성 | 저장된 템플릿에서 생성 | 프로젝트 생성 모달 → 템플릿 선택 |

### 4.4 프로젝트 데이터 모델 (projects 테이블)

| 필드 | 타입 | 설명 | 제약조건 |
|------|------|------|---------|
| id | INTEGER, PK | 프로젝트 ID | AUTOINCREMENT |
| user_id | INTEGER, FK | 생성자 | users.id, NOT NULL |
| team_id | INTEGER, FK | 팀 소속 | teams.id, NULLABLE |
| client_id | INTEGER, FK | 발주처 | clients.id, NULLABLE |
| project_name | VARCHAR(500) | 프로젝트명 | NOT NULL |
| project_type | VARCHAR(20) | 유형 | outsourcing / internal / maintenance |
| status | VARCHAR(20) | 상태 | 4.5절 참조 |
| description | TEXT | 설명 | NULLABLE |
| start_date | DATE | 시작일 | NULLABLE |
| end_date | DATE | 종료일 | NULLABLE |
| total_duration_days | INTEGER | 총 기간(일) | NULLABLE |
| contract_amount | VARCHAR(200) | 계약 금액 | NULLABLE |
| payment_method | VARCHAR(500) | 지급 방식 | NULLABLE |
| schedules | JSON | 추진 일정 배열 | NULLABLE |
| milestones | JSON | 마일스톤 | NULLABLE |
| report_opt_in | BOOLEAN | AI 정기보고 활성화 | DEFAULT FALSE |
| report_frequency | VARCHAR(10) | 보고 주기 | daily/weekly/monthly, NULLABLE |
| created_at | DATETIME | 생성일 | UTC |
| updated_at | DATETIME | 수정일 | UTC |

| 상태 | 코드 | 배지 색상 | 설명 |
|------|------|---------|------|
| 대기 | pending | 회색 | 시작 전 |
| 진행중 | in_progress | 파란색 | 작업 중 |
| 완료 | completed | 초록색 | 완료됨 |

**확장 상태** (is_client_facing = true인 업무):

| 상태 | 코드 | 배지 색상 | 설명 |
|------|------|---------|------|
| 보고 발송됨 | report_sent | 보라색 | 완료 보고 이메일 발송 후 |

| 확인됨 | confirmed | 진한 초록 | 발주처가 "확인" 처리 |

**상태 전이 규칙:**
```
[일반 업무]

[발주처 대면 업무]
  대기 → 진행중 → 완료 → 보고 발송됨 → 피드백 대기
                                              ├─ 확인됨
                                              └─ 수정 요청 → 진행중 (재작업)

[자동 확인]
  피드백 대기 상태에서 7일 무응답 → 자동 "확인됨" 전환
```

### 6.5 파일 업로드 및 AI 분석

**지원 파일 형식:**

| 형식 | 확장자 | 처리 방식 |
|------|--------|---------|
| PDF (텍스트) | .pdf | PyMuPDF로 텍스트 추출 → Gemini 텍스트 분석 |
| PDF (스캔) | .pdf | 페이지별 이미지 렌더링 → Gemini 멀티모달 분석 |

| 문서 유형 | AI 추출 대상 |

## 7. Google 스프레드시트 연동

### 7.1 개요

견적서를 Google Sheets로 작성/관리한다. 기존 업무 습관을 유지하면서, 서비스가 데이터를 활용하는 구조이다.

### 7.2 연동 방식

- **Google Sheets API v4** 사용
- OAuth2 인증 (기존 Google OAuth 확장, 추가 scope: `https://www.googleapis.com/auth/spreadsheets`)
- 사용자의 Google Drive에 시트를 생성하거나 기존 시트를 연결

### 7.3 기능 상세

| 기능 | 설명 |
|------|------|
| 시트 생성 | "견적서 만들기" → 기본 템플릿으로 새 Google Sheet 생성 |
| 시트 연결 | 기존 Google Sheet URL 입력 → 연결 |
| 임베드 보기 | 서비스 내에서 iframe으로 시트 읽기 전용 표시 |
| 편집 링크 | "Google Sheets에서 편집" → 새 탭에서 열기 |
| AI 파싱 | "견적 분석" → 시트 내용을 읽어 항목/금액/일정 구조화 |
| 동기화 | 시트 수정 시 프로젝트 금액/항목 자동 갱신 (5분 간격 폴링) |
| PDF 변환 | 발주처 발송 시 PDF로 자동 변환 |

### 7.4 AI 파싱 결과 구조

```json

  "estimate_items": [

      "quantity": 1,
      "unit": "식",
      "unit_price": 5000000,
      "amount": 5000000,
      "estimated_days": 14
    }
  ],
  "total_amount": 50000000,
  "estimated_duration_days": 120
```

### 7.5 API 엔드포인트

## 9. 완료 보고 및 이메일 발송

### 9.1 개요

발주처 대면 업무(is_client_facing=true) 완료 시, 발주처에게 이메일로 보고하여 **업무 완료 근거를 남기고 피드백을 수집**한다.

### 9.2 완료 보고 작성 흐름

1. 업무를 "완료"로 변경
2. 발주처 대면 업무인 경우 "완료 보고 작성" 모달 자동 표시
3. 모달 구성:
   - **수신자:** 발주처 담당자 이메일 (clients.contact_email에서 자동 채움)
   - **참조(CC):** 추가 이메일 입력 가능
   - **제목:** AI 자동 생성 (편집 가능). 형식: `[프로젝트명] "업무명" 완료 안내`
   - **본문:** AI 자동 생성 (업무명, 처리내용, 산출물 기반). 사용자 편집 가능
   - **첨부파일:** 업무 산출물 자동 첨부 + 추가 파일 첨부 가능
   - **발송 옵션:** "지금 발송" / "예약 발송" (날짜/시간 선택)

### 9.3 이메일 템플릿

- HTML 기반 이메일 (Jinja2 템플릿)
- 하단에 피드백 링크 자동 포함 (10장 연결)
- 포함 내용: 프로젝트명, 업무명, 완료일, 담당자, 처리 내용 요약, 첨부 파일 목록
- 회사 로고/서명 커스터마이징 (설정에서 관리)

### 9.4 완료 보고 데이터 모델 (completion_reports 테이블)

## 10. 클라이언트 피드백 시스템

### 10.1 개요

완료 보고 이메일에 포함된 피드백 링크를 통해, 발주처 담당자가 **로그인 없이** 업무 완료를 확인하고 피드백을 남길 수 있다.

### 10.2 피드백 페이지

**URL:** `/feedback/{token}`

**토큰 검증:**
- 유효기간: 발송 후 30일
- 1회 이상 사용 가능 (수정 요청 후 재피드백 허용)

**페이지 구성:**
- 상단: 프로젝트명, 업무명, 완료일, 담당자
- 중앙: 보고 내용 (본문) + 첨부파일 다운로드 링크
- 하단: 피드백 버튼 3개

| 버튼 | 동작 | 업무 상태 변경 |
|------|------|-------------|
| 확인 완료 | 확인 메시지 표시 | → confirmed |
| 수정 요청 | 수정 사유 입력 textarea 표시 | → revision_requested |
| 의견 남기기 | 코멘트 입력 textarea (상태 변경 없음) | 변경 없음 |

### 10.3 피드백 데이터 모델 (client_feedbacks 테이블)

| 필드 | 타입 | 설명 | 제약조건 |
|------|------|------|---------|
| id | INTEGER, PK | 피드백 ID | AUTOINCREMENT |
| completion_report_id | INTEGER, FK | 연관 완료 보고 | completion_reports.id |
| task_id | INTEGER, FK | 대상 업무 | tasks.id |
| feedback_type | VARCHAR(20) | 피드백 유형 | confirmed/revision/comment |
| content | TEXT | 피드백 내용 | NULLABLE |
| client_name | VARCHAR(100) | 작성자명 (선택) | NULLABLE |
| ip_address | VARCHAR(50) | 접속 IP | 감사 추적용 |
| created_at | DATETIME | 작성일 | UTC |

### 10.4 자동 확인 (7일 무응답)

- 피드백 대기(feedback_pending) 상태에서 **7일 경과** 시 자동 "확인됨(confirmed)" 전환
- Background task: 1일 1회 실행
- 자동 확인 **3일 전** 리마인더 이메일 발송 (발주처에게)
- 자동 확인 시 activity_log에 기록: "시스템에 의해 자동 확인 처리됨"

### 10.5 피드백 알림

| 이벤트 | 알림 대상 |
|--------|---------|
| 피드백 접수 (확인) | 업무 담당자 |
| 피드백 접수 (수정 요청) | 업무 담당자 + 프로젝트 멤버 전체 |
| 피드백 접수 (코멘트) | 업무 담당자 |

### 10.6 API 엔드포인트

| 메서드 | 경로 | 설명 | 인증 |
|--------|------|------|------|
| GET | `/feedback/{token}` | 피드백 페이지 (HTML) | 토큰 |
| POST | `/api/v1/feedback/{token}` | 피드백 제출 | 토큰 |
| GET | `/api/v1/tasks/{id}/feedbacks` | 업무의 피드백 이력 | 세션 |

## 11. AI 보고서

### 11.1 정기 보고서 (일간/주간/월간)

**대상:** `report_opt_in=true`인 프로젝트

**설정 화면:**

| 설정 항목 | 옵션 |
|----------|------|
| 보고 활성화 | ON / OFF (기본: OFF) |
| 주기 | 일간 / 주간 / 월간 |
| 주간 기준일 | 금요일 (기본) |
| 월간 기준일 | 마지막 영업일 (기본) |
| 발송 방식 | 검토 후 발송 (기본) / 자동 발송 |

**생성 프로세스:**
1. Cron job이 해당 주기에 트리거
2. 해당 기간 내 데이터 수집: 완료 업무, 산출물, 피드백 현황, 진행중 업무
3. GeminiService.generate_periodic_report() 호출
4. 초안 생성 → 사용자에게 알림 ("보고서 초안이 생성되었습니다")
5. 사용자가 검토/편집 후 "발송" 또는 "삭제"

**보고서 내용:**
- 기간 요약
- 완료 업무 목록 + 산출물
- 클라이언트 피드백 현황
- 진행중 업무 현황 + 진행률
- 다음 기간 예정 업무
- 이슈/지연 사항

### 11.2 프로젝트 완료 보고서 (결과 보고서)

**자동 생성 트리거 조건:**
- 프로젝트에 계약서 또는 견적서가 첨부되어 있음
- 모든 Task가 "완료(completed)" 또는 "확인됨(confirmed)" 상태
- 자동 생성 후 사용자에게 알림. 발송은 사용자가 결정

**포함 내용:**
- 프로젝트 개요 (발주처, 계약기간, 금액)
- 단계별 수행 내역
- 전체 업무 완료 현황
- 산출물 목록 (제출일, 확인일)
- 일정 준수율 (계획 대비 실적)
- 클라이언트 피드백 이력 통계
- 수금 현황
- 특이사항 (AI 요약)

| 중도금 | interim | 중간 시점 지급 |
| 잔금 | final | 프로젝트 완료 시 지급 |
| 마일스톤 | milestone | 특정 업무/단계 완료 시 지급 |

### 13.4 결제 상태

| 상태 | 코드 | 설명 |
|------|------|------|
| 대기 | pending | 아직 청구 전 |
| 청구완료 | invoiced | 세금계산서/청구서 발행 |
| 입금완료 | paid | 입금 확인 |
| 연체 | overdue | 기한 초과 미입금 |

**상태 전이:** `pending → invoiced → paid` 또는 `pending → overdue → paid`

## 14. 프로젝트 템플릿 및 반복 업무

### 14.1 프로젝트 템플릿

기존 프로젝트의 구조(업무 목록, 일정)를 템플릿으로 저장하여 재활용한다.

**데이터 모델 (project_templates 테이블):**

| 필드 | 타입 | 설명 | 제약조건 |
|------|------|------|---------|
| id | INTEGER, PK | 템플릿 ID | AUTOINCREMENT |
| user_id | INTEGER, FK | 생성자 | users.id |
| team_id | INTEGER, FK | 팀 소속 | teams.id, NULLABLE |
| name | VARCHAR(200) | 템플릿명 | NOT NULL |
| project_type | VARCHAR(20) | 프로젝트 유형 | NOT NULL |
| description | TEXT | 설명 | NULLABLE |
| task_templates | JSON | 업무 템플릿 | 아래 구조 참조 |
| schedule_templates | JSON | 일정 템플릿 | NULLABLE |
| created_at | DATETIME | 생성일 | UTC |

**task_templates JSON 구조:**
```json
[
  {
    "task_name": "요구사항 분석",
    "phase": "분석",
    "relative_due_days": 14,
    "priority": "높음",
    "is_client_facing": true
  }
]
```

**사용:** 템플릿 선택 → 시작일 입력 → `relative_due_days`로 실제 마감일 계산 → 프로젝트 + 업무 일괄 생성

| task_name | VARCHAR(300) | 업무명 | NOT NULL |
| description | TEXT | 설명 | NULLABLE |

## 16. 클라이언트 포털

### 16.1 개요

발주처 담당자가 **로그인 없이** 프로젝트 진행 현황을 직접 확인할 수 있는 외부 페이지.

### 16.2 포털 페이지

**URL:** `/portal/{portal_token}`

**표시 내용:**
- 프로젝트명, 기간, 전체 진행률 (완료 업무 / 전체 업무)
- 업무 목록 (업무명, 상태, 마감일만 표시 — 내부 메모/댓글 비공개)
- 산출물 다운로드 (완료 보고에 첨부된 파일)
- 피드백 대기 항목 (직접 피드백 가능)
- 보고서 열람 (발송된 보고서)

### 16.3 포털 토큰 데이터 모델 (portal_tokens 테이블)

| 필드 | 타입 | 설명 | 제약조건 |

| client_id | INTEGER, FK | 발주처 | clients.id |
| project_id | INTEGER, FK | 대상 프로젝트 | projects.id |
| token | VARCHAR(64) | 접근 토큰 | UNIQUE |
| expires_at | DATETIME | 만료일 | 프로젝트 종료일 + 30일 |
| is_active | BOOLEAN | 활성 여부 | DEFAULT TRUE |

- 발주처별/프로젝트별 1개 토큰
- 관리자가 언제든 토큰 재발급/폐기 가능

## 17. 캘린더 연동

### 17.1 개요

### 17.2 Google Calendar 연동

- **API:** Google Calendar API v3
- **OAuth scope:** `https://www.googleapis.com/auth/calendar`
- **동기화 대상:** 마감일(due_date)이 있는 업무
- **이벤트 형식:** 제목 `[CS] {업무명}`, 설명에 프로젝트명/우선순위 포함
- **양방향:** 캘린더에서 일정 변경 시 업무 마감일 갱신

### 17.3 Outlook 연동

- **API:** Microsoft Graph API
- **OAuth:** Azure AD
- **동일한 동기화 방식**

| refresh_token | TEXT | 갱신 토큰 | AES-256 암호화 |
| calendar_id | VARCHAR(200) | 연동 캘린더 ID | NOT NULL |
| is_active | BOOLEAN | 활성 여부 | DEFAULT TRUE |

## 26. 마이그레이션 전략 (v1 → v2)

### 26.1 데이터베이스 마이그레이션

Alembic을 사용한 단계적 마이그레이션:

| 단계 | 작업 | 데이터 영향 |
|------|------|-----------|
| 1 | 신규 테이블 생성 (clients, tasks, documents 등) | 없음 |
| 2 | contracts 테이블에 신규 필드 추가 (client_id, project_type 등) | 없음 |
| 3 | contracts → projects 테이블 리네임 | 기존 데이터 보존 |
| 4 | contracts.client (문자열) → clients 테이블 자동 생성 + project.client_id 연결 | 기존 발주처명으로 clients 레코드 생성 |
| 5 | contracts.tasks (JSON) → tasks 테이블 데이터 이전 | JSON → 독립 레코드 변환 |
| 6 | comments.contract_id → project_id 리네임 | FK 참조 변경 |
| 7 | activity_logs.contract_id → project_id 리네임 | FK 참조 변경 |
| 8 | 기존 JSON 필드 제거 (최종 정리) | tasks JSON 삭제 |

### 26.2 API 호환성

- v1 엔드포인트 (`/api/v1/contracts/*`): deprecated 표시, 일정 기간 유지 후 제거
- v2 엔드포인트: `/api/v1/projects/*`, `/api/v1/tasks/*`, `/api/v1/clients/*` 등 신규 경로
- 프론트엔드 점진 전환

### 26.3 프론트엔드 마이그레이션

- 기존 index.html + main.js를 모듈화된 컴포넌트로 분리 권장
- 기존 대시보드 코드를 프로젝트/업무 독립 뷰로 리팩토링
- Alpine.js 컴포넌트를 파일별로 분리
